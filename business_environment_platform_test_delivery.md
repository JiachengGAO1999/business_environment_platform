# 全球营商环境智能分析平台 Demo 测试与交付文档

## 1. 交付目标

本交付文档用于约束初版 demo 网站的测试、验收与交付范围。初版交付不以生产可用为目标，而以“可演示、可解释、可扩展”为目标。

交付成功的标准：

1. 网站可以本地启动并稳定浏览。
2. 核心页面完整，主要交互可用。
3. mock 数据能够支撑完整业务演示。
4. 报告结构符合营商环境专家分析框架。
5. 关键判断能够看到证据来源。
6. 审核流程可以模拟。
7. 项目结构清晰，便于后续接入真实数据和智能体 API。

## 3. 本地启动验收

### 3.1 安装依赖

要求：

- 按 README 执行安装命令应成功。
- 不应依赖开发者本机私有路径。
- 不应要求真实 API Key 才能启动 demo。

验收命令示例：

```bash
npm install
npm run dev
```

或：

```bash
pnpm install
pnpm dev
```

如果包含后端：

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

### 3.2 构建检查

前端必须通过构建：

```bash
npm run build
```

如果使用 TypeScript，必须无类型错误。

```bash
npm run typecheck
```

如果配置 lint：

```bash
npm run lint
```

## 5. 报告质量测试

### 5.1 结构检查

每份完整报告至少应包含：

- 摘要
- 宏观经济
- 产业政策
- 贸易环境
- 投资机会
- 经营风险
- 政治与地缘风险
- 企业行动建议
- 证据来源
- 待专家复核问题

### 5.2 内容检查

| 编号 | 检查项 | 合格标准 |
|---|---|---|
| Q-001 | 是否只是新闻摘要 | 不合格。必须包含分析判断 |
| Q-002 | 是否机会风险兼顾 | 合格报告必须同时写机会和风险 |
| Q-003 | 是否有行动建议 | 必须给出企业层面的建议 |
| Q-004 | 是否区分事实与判断 | 事实来自资料，判断体现模型/专家分析 |
| Q-005 | 是否有不确定性标记 | 证据不足处必须标记待复核 |
| Q-006 | 是否存在无来源数字 | 不允许出现无来源的具体数字 |
| Q-007 | 是否有过度确定预测 | 避免“必然”“一定”等绝对表述 |

### 5.3 证据链检查

每个关键结论应满足：

- 至少关联 1 条来源资料。
- 来源资料包含标题、机构、日期。
- 有对应证据片段。
- 证据片段与结论存在明显关系。
- 若证据不足，应标记为低置信度或待复核。

错误示例：

```text
巴西未来将成为全球最重要的半导体制造中心。
```

如果没有强证据支撑，这种句子不应出现。

合格示例：

```text
巴西再工业化政策可能为半导体和高附加值制造领域带来政策窗口，但短期内仍需评估本地供应链、税制和融资成本约束。
```

## 6. 数据质量测试

### 6.1 Mock 数据完整性

检查项：

- 每条资料有唯一 ID。
- 每条资料有标题、来源、国家、日期、摘要。
- 每条证据片段能关联到有效资料。
- 每份报告能关联到有效证据。
- 国家卡片中的统计数字与 mock 数据基本一致。

### 6.2 维度覆盖

mock 数据必须覆盖以下维度：

- 宏观经济
- 产业政策
- 贸易环境
- 市场机会
- 经营风险
- 政治与地缘风险
- 企业行动建议

每个维度至少有 2 条资料或证据片段。

### 6.3 低质量资料

需要包含少量低相关或噪声资料，以展示筛选能力。

示例：

- 泛泛新闻
- 重复新闻
- 企业宣传稿
- 与营商环境关系较弱的社会新闻

## 7. 真实来源抓取/解析 Smoke Test

该测试用于验证 demo 不只依赖 mock 数据。它不是生产级爬虫验收，不要求长期稳定运行，但要求能展示真实网页或 PDF 资料从获取到分析的最小闭环。

### 7.1 测试来源

建议至少选择 5 条真实资料，覆盖 2 个以上国家。可选来源包括：

| 类型 | 示例 |
|---|---|
| 国际组织数据 | World Bank country data / country overview |
| 宏观经济报告 | IMF country page / Article IV |
| 贸易政策 | WTO Trade Policy Review |
| 投资促进机构 | Invest Saudi / Invest UAE / Ministry of Investment |
| 央行报告 | Central Bank report / market expectations / quarterly review |
| 政府经济部门 | Ministry of Economy / official business environment page |

### 7.2 抓取验收

| 编号 | 测试内容 | 预期结果 |
|---|---|---|

| C-003 | 抓取 PDF | 能提取可用于分析的文本，或给出明确失败原因 |

### 7.3 解析质量验收

| 编号 | 测试内容 | 预期结果 |

| P-005 | 证据片段 | 能抽取 1-3 条支撑判断的短片段 |
| P-006 | 置信度 | 对解析不完整或证据不足的资料标记低置信度 |

### 7.4 真实资料报告验收

| 编号 | 测试内容 | 预期结果 |
|---|---|---|
| G-001 | 用真实资料生成报告段落 | 生成段落能引用真实来源 |
| G-002 | 证据点击 | 能从报告跳转到真实资料片段 |
| G-003 | 资料不足 | 报告明确标记“资料不足/需专家复核” |

### 7.5 合规与礼貌抓取

要求：

- 不做高频抓取。
- 设置超时和错误处理。
- 尊重网站访问限制。
- 不绕过登录、验证码、付费墙和反爬机制。
- Demo 中可优先使用官方公开页面、PDF、RSS 或手动导入样本。

## 8. UI/视觉验收

### 8.1 总体要求

- 页面第一屏就是可用工作台，不是宣传页。
- 桌面端布局稳定。
- 文本不重叠、不截断。
- 表格不横向溢出或有合理滚动。
- 风险标签、状态标签颜色一致。
- 证据侧栏和报告正文关系清楚。
- 加载、空状态、错误状态存在。

### 8.2 不建议出现

- 大面积装饰渐变。
- 纯营销式 hero。
- 过多动画。
- 与营商环境无关的插画。
- 没有数据密度的空泛页面。
- 只能看不能操作的静态海报式 demo。

## 9. 可维护性检查

### 9.1 代码结构

合格标准：

- 页面、组件、数据、service、类型定义分离。
- mock 数据集中管理。
- 不在多个页面重复硬编码同一份数据。
- 有清晰的 `types` 或 `schemas`。
- 有 `AgentClient` 或类似抽象，后续可替换真实 API。

### 9.2 Adapter 检查

必须能看出后续如何替换真实智能体 API：

```text
MockAgentClient -> HttpAgentClient
```

不合格情况：

- 所有 AI 结果直接写死在页面 JSX 中。
- 没有 service 层。
- 真实 API Key 写在前端代码中。

## 10. 安全检查

Demo 阶段至少检查：

- `.env` 未提交真实密钥。
- `.env.example` 存在。
- 前端没有硬编码真实 API Key。
- 外部链接使用安全打开方式。
- mock 数据不包含敏感客户信息。
- README 明确说明数据为 demo/mock。

## 11. 性能检查

Demo 阶段基础要求：

- 首页加载无明显卡顿。
- 资料库 30-100 条 mock 数据下筛选流畅。
- 报告页面滚动流畅。
- 不引入明显过重且无必要的依赖。

可选指标：

- 本地首屏加载小于 3 秒。
- 构建产物无异常膨胀。

## 12. 演示脚本

建议按以下路径演示：

### 12.1 开场

说明平台目标：

> 这是一个面向企业出海的全球营商环境智能分析工作台。它每天汇聚公开资料，筛选出与营商环境相关的政策、经济、贸易和风险信息，并生成可溯源、可审核的国别分析报告。

### 12.2 工作台

展示：

- 今日新增资料数。
- 高风险预警。
- 巴西、沙特、阿联酋等国家卡片。
- 今日重点机会和风险。

### 12.3 资料库

操作：

- 筛选“巴西”。
- 筛选“高相关”。
- 打开一条产业政策资料。

讲解重点：

- 系统不是简单堆新闻，而是给资料打上营商环境分析维度。

### 12.4 智能筛选

操作：

- 展示一条资料为何被判为高相关。
- 展示判断理由和证据片段。
- 点击“加入今日报告”。

讲解重点：

- 平台把模型判断拆成可审核的中间结果。

### 12.5 国别报告

操作：

- 打开巴西报告。
- 展示摘要、宏观压力、投资机会、经营风险、行动建议。
- 点击证据编号，查看来源片段。

讲解重点：

- 报告参考区域国别专家的结构，不是普通摘要。
- 每个关键判断都有来源支撑。

### 12.6 审核中心

操作：

- 对一段报告标记“证据不足”。
- 对另一段点击“通过”。
- 提交整体审核。

讲解重点：

- 平台定位是专家增强工具，不替代专家。

## 13. 验收分级

### 13.1 A 级：优秀 demo

- 所有核心页面完整。
- 视觉专业，有真实产品感。
- 报告结构完整，接近专家分析框架。
- 证据链清楚。
- 审核流程可操作。
- 代码结构清晰，后续易扩展。

### 13.2 B 级：可接受 demo

- 核心页面基本完整。
- 页面可运行。
- mock 数据能支撑演示。
- 报告和证据链基本可见。
- 部分交互较浅，但不影响主流程。

### 13.3 C 级：需返工

出现以下任一情况，应返工：

- 网站无法启动。
- 只有静态宣传页，没有工作台。
- 没有资料筛选和证据链。
- 报告只是普通新闻摘要。
- 页面大量错位、重叠、白屏。
- 代码完全硬编码，无法扩展。

## 14. 后续迭代建议

初版 demo 完成后，建议按以下顺序迭代：

1. 接入真实资料导入接口。
2. 引入真实 LLM/Agent API。
3. 建立报告生成 pipeline 的中间结果落库。
4. 增加真实导出 Word/PDF。
5. 增加国家/行业模板。
6. 增加多语言资料翻译与原文对照。
7. 增加报告版本管理。
8. 增加专家协同审核。
9. 引入向量检索和跨资料证据聚合。
10. 对复杂报告生成流程评估是否引入 LangGraph。

## 15. 给 Claude Code / DeepSeek 的实现提醒

请优先保证 demo 的产品闭环，而不是追求复杂 agent 框架。

实现优先级：

1. 页面结构和产品体验。
2. mock 数据质量。
3. 报告结构和证据链。
4. 审核交互。
5. 技术扩展接口。

暂时不要：

- 不要把所有内容写成营销首页。
- 不要把 AI 结果全部硬编码在 JSX 中。
- 不要引入过重的 agent 框架作为全局依赖。
- 不要把报告写成没有来源的空泛文章。
- 不要在 demo 阶段过度追求真实爬虫。
